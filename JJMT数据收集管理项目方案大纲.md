# JJMT 经销商数据收集管理项目方案大纲

> 用途：本大纲用于后续拆分为 2:1 比例（1920 × 960）的 GPT image2 方案说明图，并直接进入 PPT。视觉风格沿用《JJMT GPT image2 生图风格规范》：浅色商务、蓝色主视觉、橙色强调关键节点、红色仅用于风险/终止/异常。

## 0. 项目背景与方案定位

JJMT 中国需要对渠道内经销商的进、销、存数据进行持续收集、校验、核验和异常闭环。渠道覆盖 JJMT 到最终医院之间的完整销售通路，包括总代、二级经销商、物流配送商、医院采购代理机构等角色。

项目关键规模与约束：

- 产品规模：常用产品 5000 个以上。
- 唯一码规模：每年需要追踪约 1200 万个唯一码。
- 经销商规模：渠道内 6000 家以上经销商。
- 数据要求：及时、准确、完整。
- 销售数据：实际业务发生后 2 个工作日内上报。
- 出入库数据：每月及时上传产品条目信息。
- 库存数据：月末盘点并核对系统即时库存与盘点库存差异。
- 准确率要求：经销商数据上传准确率不低于 95%。

本项目的推荐定位不是“单纯开发一个系统”，也不是“完全依赖人工催收服务”，而是：

**用系统平台承接数据、规则、流程和证据链，用运营服务承接经销商沟通、异常推动和人工复核，用 AI/自动化能力提升数据识别、异常发现和处理效率。**

整体方案建议命名为：

**JJMT 经销商数据收集与核验运营平台**

核心闭环：

```text
多方式数据采集
→ 系统规则校验
→ 自动异常识别
→ 经销商任务推送
→ 运营服务跟进
→ 销量/库存/发票核验
→ 申诉与人工复核
→ 规则沉淀与报表复盘
```

## 0.1 公开资料补充结论

以下结论用于补充方案设计，不作为对某一供应商能力的背书。

1. 医药行业 DDI 通常指 **Distributors Data Integration**，即渠道数据自动采集。公开资料中常见做法是在经销商 PC 或系统环境中安装数据抓取与传输软件，定时从经销商系统中获取库存、购进、销售流向数据。参考：[搜狐医药行业 DDI 介绍](https://www.sohu.com/a/383312877_100121143)。
2. 公开医药流向系统资料显示，行业方案通常同时支持 DDI 直连、网页抓取、原始数据导入/API/Web Service/FTP 等多种采集方式，并结合主数据、异名库和智能清洗完成标准化。参考：[药销易医药流向管理系统](https://www.yaoxiaoyi.com/flow-manage.html)。
3. 渠道数据服务商通常把“采集、清洗、匹配、申诉、销量计算”作为一体化链路处理，同时提供渠道审计、数据处理和系统能力。参考：[阙天渠道数据管理](https://www.qtdatas.com/)。
4. 医疗器械唯一标识 UDI 是医疗器械全生命周期管理的重要基础，国家药监局已发布《医疗器械唯一标识系统规则》。参考：[国家药监局 UDI 系统规则公告](https://udi.nmpa.gov.cn/toDetail.html?CatalogId=2&infoId=41)。
5. GS1 医疗健康标准中，UDI/GS1 DataMatrix 可承载产品标识、批号、有效期、序列号等信息，适合支撑扫码追溯、库存控制和供应链透明化。参考：[GS1 Healthcare UDI](https://www.gs1.org/industries/healthcare/udi)、[GS1 Healthcare Standards](https://www.gs1.org/industries/healthcare/standards)。
6. 发票验真可参考国家税务总局全国增值税发票查验平台能力，公开页面说明支持查验近 5 年内增值税发票管理系统或电子发票服务平台开具的发票，并要求录入发票代码/号码、开票日期、金额等要素。参考：[全国增值税发票查验平台](https://inv-veri.chinatax.gov.cn/)。

## 0.2 本版方案补充假设

根据当前项目口径，本方案按以下假设展开：

- 技术栈：数据库使用 MS SQL Server；后端使用 .NET Core；Web 前端使用 Vue；应用服务器使用 IIS；移动端使用 Uni App 架构。
- 部署环境：基于 Azure Cloud 设计生产环境、测试环境和网络架构，并提供负载均衡方案。
- 用户访问：用户端同时支持 Web 浏览器访问和移动端访问；移动端形态可为微信小程序、企业微信应用或 H5 网站。
- 发票能力：发票验真和发票全票面信息获取通过第三方金税系统接口实现。
- 数据迁移：现有存量系统数据在上线前通过数据初始化导入；历史数据质量目前暂无明确评价标准，需在初始化阶段建立质量评估和问题分级。
- DDI 覆盖：约 6000 家经销商中预计 200 家使用 DDI，一般为业务量较大的经销商；预计 DDI 经销商贡献约 50% 上报量，该比例为当前预估值。
- 申诉流程：销量申诉主要针对销量与发票不匹配场景，由经销商发起，葡萄城运维人员/JJMT 渠道部人员审批，不设计复杂多级审批。
- 申诉类型：包括开票产品规格型号申诉、产品单位转换率申诉、购方/销方申诉。
- 唯一码口径：JJMT 唯一码先不等同于 UDI；系统销量核验需同时支持唯一码级核验和批号级（非唯一码）核验。
- 内部系统对接：当前仅考虑与 JJMT Data Warehouse 系统对接。
- 合同考核：上线后服务 SLA 是否写入合同考核指标，本版方案暂不展开。

# 1. 项目范围

## 1.1 总体范围定义 [[JJMT数据收集管理项目方案大纲#P02 项目范围总览图]]

项目范围建议分为两层：

### A. 系统范围

系统负责“收、验、管、推、查、报”：

- 收：支持 DDI 数据直连、销量/发票上报、扫码报量、Excel 导入、附件上传。
- 验：自动校验格式、主数据、渠道关系、发票、销量、库存、唯一码、重复数据、红冲/退货/调整。
- 管：管理经销商、渠道层级、医院采购代理机构备案、产品主数据、库位、用户角色、权限、任务。
- 推：对逾期、异常、未闭环事项自动推送提醒和升级。
- 查：提供数据追溯、发票证据链、申诉记录、人工复核记录、操作日志。
- 报：输出周报、月报、服务报告、DDI 运行报告、销量核验报告、库存差异报告。

### B. 服务范围

服务负责“培训、催收、复核、协调、复盘”：

- 经销商上线与培训。
- 数据收集日常运营和逾期催收。
- 异常数据沟通与修正推动。
- 发票校验人工复核。
- 销量申诉 2 个工作日内响应。
- 年度盘点/VBP 产品盘点支持。
- DDI 对接上线与运行监控。
- 经销商咨询、投诉和现场支持。
- 定期数据质量复盘与规则优化建议。

## 1.2 系统 + 服务衔接机制

系统与服务应形成清晰分工：

| 环节 | 系统承担 | 服务承担 | 输出结果 |
|---|---|---|---|
| 数据收集 | 多入口采集、模板控制、接口接入、扫码记录 | 指导经销商选择上报方式，推动按期上报 | 数据及时到达 |
| 数据校验 | 格式、主数据、渠道关系、发票、唯一码、库存逻辑校验 | 解释异常原因，推动经销商修正 | 数据准确可用 |
| 销量核验 | 发票、出库单、客户、产品、数量、日期、红冲规则匹配 | 对疑难单据人工复核，必要时提交 JJMT 确认 | 销量可信 |
| 库存核验 | 系统库存、月末盘点、库位和唯一码差异比对 | 协助经销商解释差异，推动补报/调整 | 库存闭环 |
| 申诉流程 | 申诉发起、证据上传、任务流转、历史规则沉淀 | 2 个工作日响应，复核并反馈处理意见 | 异常闭环 |
| 报告复盘 | 自动生成指标、明细、趋势、排名 | 输出解释、风险与改进建议 | 管理可视 |

## 1.3 项目边界建议

### JJMT 侧提供

- 经销商清单、层级关系、联系人、区域归属。
- 产品主数据、UDI/唯一码规则、批号/型号/规格映射。
- 医院、科室、医院采购代理机构备案规则。
- 发票核验、销量核验、库存核验的业务规则和审批口径。
- 现有历史数据、未结异常、申诉历史、人工规则。
- 短信、邮件、企业微信/微信服务号等通知通道的合规授权。
- 系统部署环境、安全要求和账号权限策略。
- Data Warehouse 对接规范，包括数据范围、接口方式、同步频率、字段字典和安全要求。

### 葡萄城侧提供

- 数据收集管理平台建设。
- DDI/API/SFTP/导入/扫码等采集能力建设。
- 数据治理、规则校验、发票核验、申诉流程、报表能力建设。
- 经销商培训、数据运营、异常催收、人工复核、服务报告。
- 系统运维、监控、备份、版本升级和持续优化。

### 经销商侧配合

- 按要求提供进销存、发票、出库单、库存盘点、扫码记录。
- 配合 DDI 可行性评估、系统授权、字段映射和测试。
- 对异常数据及时修正或申诉。
- 保证上传数据与真实业务一致。

# 2. 具体功能方案

## 2.1 数据收集平台

### 2.1.1 核心模块

- 经销商门户：经销商登录、待办、上报入口、异常处理、申诉入口、培训资料。
- 数据采集中心：发票报量、出库单上传、库存上传、DDI 接入、扫码报量、Excel 导入。
- 数据治理中心：字段标准化、主数据匹配、异名库、数据去重、异常规则、数据质量评分。
- 发票审计中心：对接第三方金税系统接口，完成发票验真、全票面信息获取、购销方匹配、产品明细匹配、数量金额匹配。
- 销量核验中心：销量数据、发票、出库单、唯一码、客户和医院信息的多维匹配。
- 库存核验中心：系统库存、盘点库存、库位、唯一码、在途/报废/寄售状态的差异核验。
- 申诉处理中心：经销商申诉、证据上传、运营复核、JJMT 确认、规则沉淀。
- 通知与催办中心：站内信、邮件、短信、企业微信/微信通知、升级提醒。
- 报表看板中心：及时率、准确率、完整率、异常率、DDI 运行状态、申诉 SLA、库存差异。
- 系统管理中心：组织、角色、权限、数据范围、审计日志、规则配置。

### 2.1.2 数据质量控制

数据进入系统时应分为三道控制：

1. **入口控制**：模板、字段、格式、必填项、日期、数量、单位、票据类型。
2. **业务控制**：渠道关系、医院关系、产品主数据、库位属性、唯一码状态、红冲/退货逻辑。
3. **闭环控制**：异常自动生成任务，推送经销商修正；逾期升级至运营人员和 JJMT 对口人。

### 2.1.3 自动催办与经销商推动机制

仅发现问题不够，系统必须具备“推动修正”的机制：

- 每家经销商建立月度/周度数据任务。
- 自动计算任务状态：未开始、部分上报、待修正、待复核、已闭环。
- 自动发送提醒：T-3、T-1、T、T+1、T+3。
- 异常升级路径：经销商填报人 → 经销商负责人 → 葡萄城运营 → JJMT 区域/渠道负责人。
- 建立经销商数据健康分：及时率、准确率、完整率、异常闭环时长、申诉通过率。
- 对高风险经销商输出重点跟进清单。

## 2.2 三种数据收集方式

## 2.2.1 方式一：销量/发票上报

适用对象：

- DDI 暂不具备条件的经销商。
- 临时性、低频、特殊场景数据。
- 医院采购代理机构代报场景。

功能设计：

- 支持页面录入、Excel 导入、发票/出库单附件上传。
- 支持专票、普票、全电发票不同校验要素。
- 通过第三方金税系统接口完成发票验真并获取全票面信息，包括购方、销方、发票号、开票日期、明细产品、规格、数量、金额等；对非标准附件可辅助使用 OCR 结构化识别。
- 支持出库单与发票关联。
- 支持医院采购代理机构备案支持件上传、审批、有效期控制。
- 在备案有效期内，二级经销商代报医院采购代理机构到医院的销量可自动进入有效销量候选。

关键校验：

- 购方是否匹配下游经销商/医院/代理机构。
- 销方是否匹配上游经销商。
- 产品名称、规格、型号、数量是否匹配。
- 发票日期、出库日期是否在合理期间。
- 是否重复发票、重复出库单、重复唯一码。
- 是否存在蓝字/红字、退货、销售调整、跨月调整。

## 2.2.2 方式二：扫码报量

适用对象：

- 高值耗材、植入类产品、需要一物一码追踪的业务。
- 经销商出库、医院使用、手术消耗、退回返库等扫码场景。

功能设计：

- 移动端扫码：支持 UDI/唯一码解析，记录扫码人、时间、地点、单据、客户、业务类型。
- 扫码业务类型：总代出库、经销商出库、调拨出库、医院销售、手术消耗、退货返库、报废。
- 扫码记录与 CS 跟台信息、手术信息、医院/科室信息匹配。
- 支持两类业务路径：
  - 经销商扫码出库 vs CS 跟台信息匹配。
  - 一扫减去三扫 vs 二扫并匹配唯一码。
- 支持异常提示：唯一码不存在、已消耗、重复扫描、渠道链不匹配、库位状态异常。

关键价值：

- 用扫码数据作为发票/销量数据的辅助证据。
- 对植入类产品形成更强证据链。
- 为库存差异、退货、在途、报废提供唯一码级别追溯。

## 2.2.3 方式三：DDI 数据直连

### A. DDI 行业做法理解

医药行业 DDI 的核心不是让经销商人工整理文件，而是尽量从经销商 ERP、进销存系统、财务系统或业务数据库中自动提取指定字段，并定时传输给药械企业。

公开资料中的常见做法包括：

- 在经销商 PC 或服务器安装采集代理。
- 从经销商业务系统数据库、报表、导出文件中采集数据。
- 定时传输库存、购进、销售流向数据。
- 通过 DDI 直连、网页抓取、智能导入、API、Web Service、FTP/SFTP 等方式组合采集。
- 对采集数据进行主数据匹配、异名识别、规则清洗。
- 通过 DDI 采集看板监控采集状态、数据到达情况、采集失败原因。
- 对未到达数据进行催收提醒，并支持历史数据回滚或补采。

### B. JJMT 推荐 DDI 分层方案

建议将 DDI 分为三种模式，而不是只设计一种接口。

结合当前渠道规模，DDI 推广建议优先覆盖约 200 家业务量较大的经销商。这些经销商数量占比不高，但预计贡献约 50% 的上报量，因此 DDI 的价值重点不是覆盖所有经销商，而是优先把高频、高金额、高复杂度的数据链路自动化，降低核心数据的人工收集压力。

#### 模式 1：标准接口型 DDI

适用对象：

- 大型总代、核心二级经销商。
- 经销商具备 ERP/API/数据库接口能力。

方式：

- API、Web Service、SFTP、数据库只读视图、标准文件交换。
- 日增量 + 月末全量校验。
- 支持回滚补采 60 天或按 JJMT 规则配置补采周期。

优势：

- 稳定、自动化程度高。
- 对经销商日常人员依赖低。
- 适合核心渠道批量推广。

#### 模式 2：边缘采集代理型 DDI

适用对象：

- 经销商系统较老，无法开放标准 API。
- 但可在本地部署采集程序或读取导出文件。

方式：

- 在经销商本地部署轻量采集代理。
- 仅读取授权范围内的数据表、视图或指定目录文件。
- 本地完成初步字段映射、加密、断点续传。
- 通过 HTTPS 443 主动上送到 JJMT 平台，不开放经销商内网入站端口。

优势：

- 对老系统兼容性更好。
- 数据自动化程度高于人工导入。
- 安全边界相对清晰。

#### 模式 3：AI 辅助导入型 DDI Lite

适用对象：

- 小型经销商。
- 系统能力弱、字段不标准、只能导出 Excel/PDF/邮件附件。

方式：

- 经销商仍按固定节奏上传 Excel、CSV、PDF 或邮件附件。
- 系统使用智能字段识别、模板学习、异名匹配、OCR 辅助结构化。
- 运营人员只处理低置信度字段或异常记录。

典型场景：

某长尾二级经销商没有开放 ERP 接口的能力，每月只能从本地进销存软件导出“销售明细.xlsx”和“库存汇总.xlsx”，同时将部分出库单或发票 PDF 作为邮件附件提交。传统方式下，运营人员需要人工判断“客户名称”“商品名称”“规格型号”“批号”“数量”“库位”等列对应 JJMT 标准字段，再手工修正产品异名和单位差异。DDI Lite 的做法是让经销商继续上传原始文件，但系统自动识别文件结构并生成映射建议，运营人员只确认异常项，从而把“全量人工整理”变成“系统预处理 + 人工复核低置信度数据”。

AI 辅助的具体内容：

- 文件类型识别：识别 Excel、CSV、PDF、图片附件属于销售、库存、发票、出库单还是支持件。
- 字段语义识别：将经销商原始列名如“商品”“货品名称”“规格”“客户”“往来单位”“仓库”自动映射到 JJMT 标准字段。
- 模板学习：同一经销商多次上传后，系统记忆其文件结构和字段映射，下次自动套用。
- 产品异名匹配：对产品名称、规格型号、生产厂家等进行相似度匹配，生成标准产品候选。
- 单位换算建议：识别“盒、支、个、套”等单位差异，结合历史规则推荐转换率。
- 客户/医院异名匹配：识别医院简称、集团名称、分院名称、曾用名，生成客户匹配候选。
- 票据/附件结构化：对出库单、发票 PDF 或图片进行 OCR，并提取单号、日期、客户、产品、数量等信息。
- 异常解释草稿：对未匹配数据自动生成原因草稿，例如“产品规格疑似异名”“购方名称疑似分公司名称”“单位换算缺少规则”。
- 人工确认入口：AI 结果必须保留置信度，低置信度记录进入运营人员复核队列，确认结果沉淀为该经销商的历史规则。

优势：

- 降低推广门槛。
- 比完全人工导入更高效。
- 适合快速覆盖长尾经销商。

### C. DDI 标准字段建议

基础字段：

- 经销商编码、经销商名称。
- 客户名称、客户编码、客户类型。
- 单号、单据日期、单据类型、单据备注。
- 发票号、发票日期。
- 商品编码、产品名称、产品规格、产品型号。
- 批号、JJMT 唯一码、序列号（如有）。
- 数量、单位。
- 生产厂家。
- 库位名称、库位类型。
- 上游经销商、下游经销商、医院、科室。

扩展字段：

- 数据来源系统。
- DDI 采集批次号。
- 原始字段名和原始值。
- 映射版本号。
- 数据到达时间。
- 校验状态。

### D. DDI 实施步骤

1. 经销商分级：核心经销商、普通经销商、长尾经销商。
2. 可行性评估：系统类型、数据库、导出能力、网络、安全限制、联系人。
3. 授权与协议：数据范围、字段范围、采集频率、安全责任、运维窗口。
4. 主数据对齐：经销商、产品、客户、医院、库位、单位、异名表。
5. 字段映射：经销商原始字段 → JJMT 标准字段。
6. 接口/代理开发：API、SFTP、数据库视图、边缘代理、文件目录采集。
7. 联调测试：历史样本、增量样本、异常样本。
8. 并行运行：人工上报与 DDI 同时运行 1-2 个周期。
9. 正式上线：切换为 DDI 主采集方式。
10. 运行监控：采集心跳、数据量波动、异常率、延迟、补采任务。

### E. AI 可增强方向

AI 不建议直接替代业务确认，但可以增强效率：

- 自动识别经销商 Excel 字段含义，生成映射建议。
- 自动识别产品名称/规格异名，生成匹配候选。
- 自动发现异常数据：销量突增突降、库存负数、唯一码重复、异常客户路径。
- 自动生成经销商异常原因说明草稿。
- 根据历史申诉结果推荐本次申诉处理意见。
- 对发票/出库单附件进行 OCR 和结构化提取。

AI 处理结果应保留置信度和人工确认机制，不能无痕修改业务数据。

## 2.3 库存核验

### 2.3.1 核验目标

确保系统即时库存、经销商月末盘点库存、库位属性、唯一码状态之间一致，并能解释差异来源。

### 2.3.2 功能设计

- 月末盘点任务自动生成。
- 经销商上传盘点库存或扫码盘点。
- 支持主仓库、自有分库、寄售库、报废库、在途库等库位类型。
- 系统计算理论库存：
  - 期初库存 + 入库 - 出库 - 销售 - 退货/报废/调整 = 期末理论库存。
- 与经销商盘点库存进行比对。
- 按产品、批号、唯一码、库位、经销商维度输出差异。
- 支持差异原因分类：
  - 未报出库。
  - 未报入库。
  - 扫码缺失。
  - 在途未确认。
  - 寄售物权不一致。
  - 报废未处理。
  - 销售/退货跨月。
  - 唯一码重复或缺失。
- 支持经销商差异说明、附件上传、运营复核和 JJMT 确认。

### 2.3.3 盘点支持服务

- 年度盘点和 VBP 产品专项盘点前，提供盘点通知、模板、培训。
- 盘点期间提供热线/在线支持。
- 盘点后输出差异清单、重点经销商清单和修正建议。
- 对长期差异较大的经销商纳入重点跟进或飞检建议清单。

## 2.4 销量核验

### 2.4.1 核验目标

确保经销商上报销量与真实业务、发票、出库单、渠道关系和客户信息一致。

销量核验需要同时支持两类产品核验场景：

- **唯一码级核验**：适用于 JJMT 已按一物一码管理的产品，系统按 JJMT 唯一码追踪出库、销售、扫码、库存和消耗状态。
- **批号级核验**：适用于非唯一码或历史未完整采集唯一码的产品，系统按产品编码/规格型号、批号、数量、单位、发票和出库单进行核验。

### 2.4.2 自动核验规则

- 通过第三方金税系统接口完成发票真伪查验，并获取全票面结构化信息。
- 购方与下游经销商/医院匹配。
- 销方与上游经销商匹配。
- 产品名称、规格、型号、批号、JJMT 唯一码匹配；非唯一码产品按批号、数量和单位进行核验。
- 数量、单位、金额逻辑一致。
- 开票日期、发货日期、手术日期在合理期间。
- 医院采购代理机构备案有效期校验。
- 红字发票、销售退回、销售调整、跨月调整特殊规则。
- 重复发票、重复出库单、重复唯一码识别。

### 2.4.3 核验结果状态

- 自动通过。
- 待经销商补充。
- 待运营复核。
- 待 JJMT 确认。
- 申诉中。
- 申诉通过。
- 申诉驳回。
- 已作废/红冲。

## 2.5 申诉流程

销量申诉主要处理“经销商上报销量与发票核验结果不匹配”的情况。流程应保持轻量，由经销商发起申诉，葡萄城运维人员和/或 JJMT 渠道部人员审批确认，不设计复杂多级审批。

### 2.5.1 适用场景

- 开票产品规格型号与 JJMT 标准产品不一致，需要申诉产品规格型号匹配关系。
- 发票或出库单中的产品单位与 JJMT 标准单位不一致，需要申诉产品单位转换率。
- 购方/销方名称存在简称、曾用名、集团/分公司差异，需要申诉购方/销方匹配关系。
- 医院因开票要求导致票面名称特殊。
- 医院采购代理机构代报场景。
- 红冲、退货、销售调整导致系统规则无法自动判断。
- 特殊渠道链或临时配送关系。

### 2.5.2 流程设计

```text
系统发现未匹配/异常销量
→ 经销商发起申诉
→ 选择申诉类型：规格型号 / 单位转换率 / 购方销方
→ 上传说明和支持材料
→ 系统基于历史规则给出建议
→ 葡萄城运维人员复核
→ 必要时提交 JJMT 渠道部确认
→ 输出申诉结果
→ 结果进入历史规则库
→ 后续同类数据自动复用或半自动推荐
```

### 2.5.3 SLA 建议

- 申诉受理：系统即时受理。
- 运营响应：2 个工作日内响应。
- 一般申诉处理：3-5 个工作日闭环。
- 需 JJMT 渠道部确认事项：按渠道部确认时间单独统计。
- 高风险事项：当天标记并升级。

# 4. 系统架构建议

## 4.1 数据量估算

按当前背景估算：

- 唯一码事件：约 1200 万/年，约 100 万/月。
- 经销商：6000+。
- 产品：5000+。
- DDI 经销商：预计约 200 家，预计贡献约 50% 上报量。
- 单据与发票：取决于经销商交易频率，建议按千万级明细行/年设计。
- 附件：发票、出库单、支持件、盘点附件可能成为主要存储压力，建议按 TB 级对象存储规划。

这个规模对关系型数据库并不极端，但对“数据质量规则、异步处理、附件存储、审计日志、批量导入、DDI 稳定性”要求较高。

## 4.2 软件架构建议

建议采用 Azure Cloud 上的分层架构，技术栈按 JJMT 当前口径设计：

- Web 前端：Vue，用于经销商门户、JJMT 管理端、葡萄城服务运营工作台。
- 移动端：Uni App，用于微信小程序、企业微信应用或 H5 移动站点，支撑扫码、待办、异常处理和申诉。
- 后端服务：.NET Core Web API，承接业务接口、移动端接口、DDI 接入接口、Data Warehouse 对接接口。
- 应用承载：IIS，建议部署在 Azure Windows VM / VM Scale Set，或使用支持 .NET 的托管 Web App 形态承载。
- 数据库：MS SQL Server。优先建议 Azure SQL Managed Instance 或 Azure SQL Database；如 JJMT 要求完全自管理 SQL Server，可采用 SQL Server on Azure VM。
- 文件存储：Azure Blob Storage，用于发票、出库单、盖章支持件、盘点附件、导入原始文件。
- 缓存：Azure Cache for Redis，用于会话、热点主数据、任务状态缓存。
- 消息与异步任务：Azure Service Bus / Storage Queue + .NET Worker Service，用于批量导入、DDI 数据处理、发票核验、通知推送、报表生成。
- 发票服务：通过第三方金税系统接口完成发票验真和全票面信息获取。
- 数据治理：规则引擎、主数据匹配、异名库、单位换算、唯一码/批号核验、历史申诉规则沉淀。
- 报表与对接：系统报表中心提供运营看板，同时按约定向 JJMT Data Warehouse 同步标准化后的数据。

### 4.2.1 用户访问架构

- Web 浏览器访问：经销商、JJMT 用户、葡萄城运营人员通过 HTTPS 访问 Vue Web 应用。
- 移动端访问：经销商或运营人员通过微信小程序、企业微信应用或 H5 访问 Uni App 前端。
- 扫码场景：移动端调用扫码能力，采集 JJMT 唯一码、批号、产品、库位、客户、业务类型等信息。
- 统一身份与权限：Web 与移动端共用账号、角色、权限和数据范围控制。

### 4.2.2 业务分层

- 接入层：Web、移动端、DDI API、SFTP/文件导入、第三方金税接口、Data Warehouse 对接。
- 应用层：经销商管理、数据采集、销量核验、库存核验、发票审计、申诉流程、通知催办、报表看板。
- 规则层：字段校验、主数据匹配、渠道关系校验、发票匹配、唯一码/批号核验、库存差异规则、申诉规则。
- 数据层：MS SQL Server 业务库、Blob 附件库、Redis 缓存、消息队列、审计日志。
- 运维层：Azure Monitor、Application Insights、Log Analytics、备份恢复、安全扫描。

## 4.3 软硬件资源建议

以 Azure 生产环境为例，建议初始配置如下，后续根据 DDI 上报量、附件量、并发量和批处理窗口横向扩展。

| 层级 | Azure 建议资源 | 初始建议 | 说明 |
|---|---|---:|---|
| 入口与负载均衡 | Azure Application Gateway WAF v2 | 1 套，跨可用区 | HTTPS 入口、WAF、防护、七层负载均衡 |
| Web/API 应用服务器 | Azure Windows VM / VM Scale Set + IIS | 2 台，4-8 vCPU，16-32GB RAM | 承载 Vue 静态站点、.NET Core API、IIS 应用服务 |
| 移动端服务 | 与 Web/API 共用 API 层 | 复用应用服务器 | Uni App 通过 API 访问业务服务 |
| DDI/API 接入服务 | 独立 Windows VM / VM Scale Set + IIS | 2 台，4-8 vCPU，16-32GB RAM | 承接 DDI API、SFTP/文件导入、经销商接入 |
| 异步任务/数据处理 | Azure VM / Worker Service | 2-3 台，8 vCPU，32GB RAM | 批量导入、清洗、发票核验、通知、报表任务 |
| 数据库 | Azure SQL Managed Instance / Azure SQL Database / SQL Server on Azure VM | 16 vCPU，64-128GB RAM 起 | MS SQL Server 技术路线，按 JJMT 运维偏好选托管或自管 |
| 文件存储 | Azure Blob Storage | 初始 2-5TB，可扩展 | 发票、出库单、支持件、盘点附件、导入原始文件 |
| 缓存 | Azure Cache for Redis | Standard/Premium 起 | 热点主数据、任务状态、会话缓存 |
| 消息队列 | Azure Service Bus | Standard/Premium | 异步任务解耦、削峰填谷、失败重试 |
| 监控日志 | Azure Monitor + Application Insights + Log Analytics | 1 套 | 应用性能、接口失败、DDI 心跳、异常日志 |
| 备份恢复 | Azure Backup / SQL 自动备份 / Blob 版本管理 | 按策略配置 | 建议数据库 30-90 天备份，附件启用生命周期管理 |
| Data Warehouse 对接 | Azure Data Factory / API / SFTP | 按 JJMT 规范 | 向 JJMT Data Warehouse 同步标准化数据 |

### 4.3.1 负载均衡建议

推荐采用两层负载均衡：

- **公网七层负载均衡**：Azure Application Gateway WAF v2，统一承接 Web、移动端 API、经销商门户、管理端访问，提供 HTTPS 终止、WAF 防护、URL 路由和后端健康检查。
- **后端应用负载均衡**：应用服务器采用 2 台以上 IIS 节点，部署在不同可用区或可用性集中，由 Application Gateway 按健康检查结果分发流量。
- **DDI 接入隔离**：DDI/API 接入服务建议使用独立后端池，避免 DDI 批量上报影响普通 Web 用户访问。
- **异步削峰**：批量导入、发票核验、DDI 数据清洗不直接占用在线请求线程，进入 Service Bus 队列后由 Worker Service 异步处理。

如 JJMT 需要跨区域灾备，可进一步引入 Azure Front Door 做全球入口和区域级流量切换；本版先按单区域高可用方案设计。

## 4.4 网络架构建议

建议基于 Azure Virtual Network 进行安全分区：

- 公网入口区：Azure Application Gateway WAF v2，仅开放 HTTPS 443。
- 应用子网：IIS Web/API 应用服务器，不直接暴露公网，仅接受 Application Gateway 转发。
- DDI 接入子网：DDI/API/SFTP 接入服务，支持 IP 白名单、证书、Token、经销商级密钥。
- 异步处理子网：Worker Service、批处理任务、发票核验任务、数据清洗任务。
- 数据子网：MS SQL Server、Redis、Service Bus Private Endpoint、Blob Storage Private Endpoint，不直接暴露公网。
- 管理子网：Azure Bastion / VPN / Jump Server，运维访问启用 MFA、最小权限和操作审计。
- 对接出口：与第三方金税系统、JJMT Data Warehouse 的接口访问通过固定出口 IP、专线/VPN 或安全 API 网关按 JJMT 安全要求配置。

### 4.4.1 DDI 网络与安全建议

- 经销商侧 DDI 采集代理只主动向平台 HTTPS 443 上送数据，不要求 JJMT 或葡萄城主动访问经销商内网。
- 对 200 家 DDI 经销商建立独立接入配置，包括经销商编码、密钥、证书、IP 白名单、采集频率、字段范围。
- 支持断点续传、失败重试、本地加密缓存、重复批次识别。
- DDI 数据先进入接入区暂存和校验，再进入业务库，避免异常批次直接污染正式数据。
- DDI 运行监控包括心跳、最后成功采集时间、当日数据量、失败原因、补采状态。

### 4.4.2 数据与接口安全

- 全链路 TLS。
- 数据库和 Blob Storage 启用静态加密。
- 敏感字段脱敏展示，如联系人手机号、发票敏感信息等。
- 用户、角色、数据范围权限控制。
- 全量操作日志和数据变更日志。
- 发票、支持件、申诉材料留痕。
- 第三方金税系统接口调用记录留痕，包括请求批次、返回状态、失败原因。
- Data Warehouse 同步留痕，包括同步批次、数据范围、成功/失败记录。
- 数据库每日备份，关键数据定期恢复演练。
- 上线前完成安全扫描、接口权限检查和必要的渗透测试。

初始化步骤：

2. 数据模板下发。

4. 主数据清洗。
5. 异名库初建。
6. 渠道关系校验。
7. 期初库存确认。
8. 历史异常导入。
9. 试算与抽样复核。
10. 上线前数据冻结与最终导入。

由于当前历史数据质量暂无明确评价标准，初始化阶段建议增加“历史数据质量评估”工作：

- 建立评估维度：完整性、唯一性、一致性、及时性、可追溯性、主数据匹配率。
- 对历史销量、发票、库存、唯一码/批号、经销商和客户信息分别评分。
- 输出问题分级：可自动修复、需运营确认、需 JJMT 确认、暂不迁移但归档保留。
- 对低质量历史数据保留原始值、清洗值和清洗规则，避免上线后无法追溯。
- 对历史规则、历史申诉结果和人工确认记录优先迁移，用于新系统规则库初始化。

# 7. 当前已确认口径与待补充事项

## 7.1 当前已确认口径

1. 部署环境：按 Azure Cloud 设计系统、软硬件资源和网络架构。
2. 技术栈：MS SQL Server、.NET Core、Vue、IIS、Uni App。
3. 用户访问：支持 Web 浏览器和移动端访问，移动端可采用微信小程序、企业微信或 H5。
4. 发票能力：使用第三方金税系统接口进行发票验真并获取全票面信息。
5. 历史数据：上线前通过数据初始化导入；历史数据质量暂无明确评价标准，需要在初始化阶段进行质量评估。
6. DDI 范围：约 6000 家经销商中预计 200 家使用 DDI，预计贡献约 50% 上报量。
7. 申诉流程：主要处理销量与发票不匹配，经销商发起，葡萄城运维人员/JJMT 渠道部人员审批，不设计复杂多级审批。
8. 申诉类型：开票产品规格型号申诉、产品单位转换率申诉、购方/销方申诉。
9. 唯一码口径：JJMT 唯一码暂不等同于 UDI；销量核验同时支持唯一码级核验和批号级核验。
10. 内部系统对接：当前仅考虑与 JJMT Data Warehouse 系统对接。
11. 合同考核：上线后 SLA 是否写入合同考核指标，本版暂不展开。

## 7.2 后续仍建议补充确认

1. Azure 所在区域、订阅归属、网络命名规范、安全基线和 JJMT 运维边界。
2. 第三方金税系统接口的具体厂商、调用频率、返回字段、并发限制和异常处理规则。
3. Data Warehouse 的数据接收方式、同步频率、字段字典、历史回补范围。
4. 200 家 DDI 经销商名单、系统类型、接口能力、联系人和上线优先级。
5. 医院采购代理机构备案的有效期、审批规则、支持材料格式。
6. 批号级核验与唯一码级核验在不同产品线上的适用边界。
